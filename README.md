# ZK Promo Code Redeemer

[![ci](https://github.com/HarleysCodes/zk-promo/actions/workflows/ci.yml/badge.svg)](https://github.com/HarleysCodes/zk-promo/actions/workflows/ci.yml)

A privacy-preserving promo code redemption dApp on the Midnight Network.

**The plaintext promo code never touches the chain.** A company issues
codes by submitting a 32-byte hash of each code to a public ledger;
users claim by submitting the plaintext to a private witness inside a
ZK proof, the prover hashes it inside the circuit, and the circuit
flips a public "claimed" flag for that hash. Bots scraping the mempool
see hashes only.

## What this demonstrates

| Privacy primitive | Where it shows up |
|---|---|
- **Witness (private input)** | `user_promo_code(salt)` in `contract/src/zk_promo.compact` (memorable-code path) and `user_redeem_secret()` (high-entropy redemption path) |
| **Public commitment set** | `validCodes: Set<Bytes<32>>` ledger |
| **Per-item claimed state** | `claimed: Map<Bytes<32>, Boolean>` ledger |
| **Domain-separated hash** | `persistentHash<Vector<2, Bytes<32>>>([tag, code])` in `hashCode()` |
| **Explicit disclosure boundary** | `disclose(h)` before any ledger read/write |

This is not a counter or a bboard. There is no public mutable "current
value." The only state that's visible on-chain is the set of issued
hashes and the set of hashes that have been claimed. The linkage
between a particular user, a particular plaintext, and a particular
claim is held only inside the user's ZK proof.

## Quick start (clone → compile → see a tx on Preview)

The full reproducible path, end-to-end, in 5 commands:

```bash
# 1. Install the Compact compiler (one-time)
curl --proto '=https' --tlsv1.2 -LsSf \
  https://github.com/midnightntwrk/compact/releases/latest/download/compact-installer.sh | sh
export PATH="$HOME/.local/bin:$PATH"
compact update 0.31.0

# 2. Clone, compile the contract, install deps
git clone https://github.com/HarleysCodes/zk-promo
cd zk-promo
cd contract && ~/.local/bin/compact compile src/zk_promo.compact managed && npm install && cd ..
cd cli && npm install && cd ..

# 3. Start the local proof server (in another terminal)
docker run -d --name midnight-proof-server -p 6300:6300 \
  ghcr.io/midnightntwrk/midnight-proof-server:latest

# 4. Generate a wallet, fund via Preview faucet, deploy
cd cli
npx tsx src/wallet.ts                              # prints a seed + unshielded address
#   → go to https://faucet.preview.midnight.network, paste the unshielded address, claim tNight
#   → wait ~30s for the unshielded balance to appear
export WALLET_SEED="<paste the seed from step 4>"
npx tsx src/deploy.ts                              # writes contract-address.txt

# 5. Issue a code, claim it, check status
npx tsx src/issue.ts WINTER24                       # operator registers a hash
npx tsx src/claim.ts WINTER24                       # user redeems with plaintext
npx tsx src/status.ts                               # shows on-chain state
```

Every command uses the **real Midnight SDK** (`@midnight-ntwrk/midnight-js`,
`@midnight-ntwrk/wallet-sdk-*`, `level-private-state-provider`,
`indexer-public-data-provider`, `http-client-proof-provider`) — same
shape as `midnightntwrk/example-counter/counter-cli`.

## What's actually on-chain

After step 5 above, on Midnight Preview:

- The contract is deployed at the address printed by `deploy.ts`
  (saved to `contract-address.txt`)
- The `validCodes` set contains one hash: the canonical
  `persistentHash<Vector<2, Bytes<32>>>(["zk-promo:v1:", "WINTER24"])`
- The `claimed` map has `true` for that hash

You can verify with `npx tsx src/status.ts` — it walks the ledger via
the public indexer and prints each issued hash and its claim status.

## The CLI commands

| Command | What it does | Network effect |
|---|---|---|
| `wallet.ts` | Build or restore an HD wallet (3 sub-wallets: Shielded, Unshielded, Dust) | None |
| `deploy.ts` | Deploy the ZK Promo contract | One deploy tx |
| `issue.ts <code>` | Operator: submit the *hash* of `<code>` to the chain | One `issue` tx per code |
| `claim.ts <code>` | User: submit `<code>` via the witness; prover hashes inside the circuit | One `claim` tx |
| `status.ts` | Read `validCodes` and `claimed` from the chain via the indexer | None (read-only) |

All four `*tx` commands sign with the wallet whose seed is in the
`WALLET_SEED` env var, fund via the wallet's unshielded balance, and
land in real Preview blocks.

## Build & verify (CI / fresh-clone recipe)

```bash
# 1. Install the Compact CLI (one-time)
curl --proto '=https' --tlsv1.2 -LsSf \
  https://github.com/midnightntwrk/compact/releases/latest/download/compact-installer.sh | sh
export PATH="$HOME/.local/bin:$PATH"
compact update 0.31.0

# 2. Compile + install
cd contract && ~/.local/bin/compact compile src/zk_promo.compact managed && npm install && cd ..
cp -r contract/managed cli/managed   # mirror CI: cli/ tests import the compiled module
cd cli && npm install && cd ..

# 3. Build + run tests
cd cli && npm test
```

Expected output:

```
✓ 12 contract tests passed
  (covers hash determinism, contract shape, witness boundary)
```

The test suite verifies:

1. Hash is deterministic
2. Different codes produce different hashes
3. Hash is 32 bytes
4. Empty code rejected
5. Code > 32 bytes rejected
6. Contract instantiates with our witness
7. Each circuit method (`issue`, `claim`, `isClaimed`) is callable
8. Issue returns non-zero hash
9. Plaintext is captured at the witness boundary
10. Contract's `initialState` is exposed
11. Witness name matches `contract-info.json`
12. **Operator's `hashPromoCode()` produces the SAME 32-byte output as
    the on-chain `persistentHash<Vector<2, Bytes<32>>>(...)`** — this
    is the contract's "claim will succeed" gate. If this test ever
    fails, the operator and the contract have drifted and `claim()`
    will silently reject every redemption in production.

## Verifying the privacy property

The dApp makes one specific privacy claim: **the plaintext promo
code is not on-chain.** A reviewer can verify this in three ways:

1. **Inspect the witness declaration** in `contract/src/zk_promo.compact`:
   `witness user_promo_code(salt: Bytes<0>): Bytes<32>;` — the
   plaintext is supplied by the user's prover, not by any circuit
   argument. The `salt` parameter is `Bytes<0>` (empty) today and
   exists only as a forward-compat hook for salted codes.

2. **Inspect the public ledger fields** in the compiled
   `managed/compiler/contract-info.json`: only `validCodes: Set<...>`
   and `claimed: Map<Bytes<32>, Boolean>` are exported. There is no
   `plaintextCode` or anything similar.

3. **Read the `issue()` and `claim()` circuit bodies** in
   `zk_promo.compact`: neither one accepts a plaintext string as a
   parameter. The only string-shaped value in the contract is the
   constant `pad(32, "zk-promo:v1:")` domain tag.

## Architecture notes

- **Wallet wiring**: `cli/src/api.ts` uses the full WalletFacade +
  4 midnight-js providers stack, mirroring `example-counter/counter-cli`.
  The proof server runs locally (Docker); the indexer + RPC + WS endpoints
  hit Preview directly. The bridge between WalletFacade and
  midnight-js's `WalletProvider` interface reimplements the
  counter-cli intent-signing workaround for the wallet SDK's `pre-proof`
  bug.

- **TypeScript at the SDK boundary**: midnight-js SDK generics are tightly
  coupled to the `counter-contract` package layout. Our dapp uses its own
  generated `Contract` class, which doesn't fit those generics cleanly.
  We use `// @ts-nocheck` (api.ts) and `as any` casts at the SDK
  boundaries, leaving the rest of the code fully typed. This matches the
  pragmatic pattern counter-cli uses internally.

- **Test architecture**: 12 tests run in-process (no network) using the
  compiled contract's `Contract` class with a stubbed witness. The
  hash recipe is tested byte-for-byte against the contract's
  `persistentHash` invocation. CLI commands run against Preview; the
  in-process suite and the Preview path both use the same hash
  recipe so they can't drift.

## Single-use semantics

Promo codes (and redemption tokens, see below) are **globally
single-use**: once any user has successfully claimed a code, no other
user can claim it. The on-chain state is `claimed: Map<Bytes<32>, Boolean>`
and the `claim()` circuit asserts `claimed.lookup(h) == false` before
flipping it to true. First claimer wins, full stop. This is by design
(typical promo-code UX) but worth flagging explicitly.

## Threat model

The privacy claim — *the plaintext is not on-chain* — is true at the
network layer (no plaintext in the ledger, no plaintext in the
mempool-observable transaction bytes). It is **not** a claim about
the secret's strength against an active adversary.

**The promo path (`issue` / `claim` / `user_promo_code`) is
vulnerable to wordlist pre-image search.** The hash recipe is
public (`persistentHash([pad(32, "zk-promo:v1:"), code])`), the
domain tag is public, and `validCodes` is a public set on-chain.
An observer can:

1. Take a wordlist of likely memorable codes (`WINTER24`, `SAVE10`,
   `SUMMER25`, brand names, etc.).
2. Hash each candidate using the same public recipe.
3. Compare against `validCodes` to recover the plaintext.

Because memorable codes are low-entropy, this brute-force is cheap.
In the worst case, a watcher can crack a code as soon as the
operator issues it and submit `claim()` before the intended user.

**The redemption path (`issueRedemption` / `claimRedemption` /
`user_redeem_secret`) is not vulnerable to this attack.** The
redeemable artifact is a full 32-byte random secret, the pre-image
space is 2²⁵⁶, and brute-force is infeasible. Production
deployments should use the redemption path. The promo path is
provided for demo UX where memorable codes matter.

Other notes on the privacy model:

- **Claimer linkability.** A successful `claim()` is publicly
  visible on-chain (tx id, block height, fee paid by the claimer's
  unshielded wallet). The plaintext code is private; the act of
  claiming it is not.
- **No operator authorization yet.** Today, `issue()` can be called
  by any wallet. Adding operator-gated issuance is planned as a
  follow-up and will use an in-circuit operator commitment (not
  `ownPublicKey()`, which is witness-sourced and not a trusted
  caller identity).
- **No off-chain leak protection.** The user must keep the plaintext
  code out of any logs, screenshots, or shared screens. The contract
  protects against on-chain exposure only.
